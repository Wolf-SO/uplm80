# uplm80 Compiler Bug Fixes

## Bug 1: SHR/SHL/ROL/ROR/SCL/SCR Operations Fail for BYTE Operands

**Status**: FIXED
**Severity**: Critical
**Location**: codegen.py (_gen_builtin for SHR, SHL, ROL, ROR, SCL, SCR)

**Description**:
The shift and rotate operations assumed the first argument was in HL after
_gen_expr(), but for BYTE types the value is returned in A, not HL. The
operations would then shift garbage values from HL instead of the byte value.

**Example**:
```plm
DECLARE X BYTE;
X = 0x5A;
Y = SHR(X, 4);  /* Should be 0x05, but gets garbage */
```

**Fix applied**:
For all six shift/rotate operations (SHR, SHL, ROL, ROR, SCL, SCR), added
check for BYTE return type and move value from A to HL before proceeding:
```python
arg_type = self._gen_expr(args[0])
if arg_type == DataType.BYTE:
    self._emit("MOV", "L,A")
    self._emit("MVI", "H,0")
```

This fix was applied to both constant-shift and variable-shift code paths.


## Bug 2: ADDRESS Comparison Operators (< and >) Are Broken

**Status**: FIXED
**Severity**: Critical
**Location**: codegen.py (_emit_jump_on_false_16bit, _emit_jump_on_true_16bit, _gen_comparison)
             upeep80/peephole.py (_translate_instruction for JP opcode)

**Description**:
The `<` and `>` comparison operators did not work correctly for ADDRESS
(16-bit) types. Two issues were fixed:

1. codegen.py was using sign-based comparisons (checking sign flag) instead of
   carry-based comparisons for unsigned ADDRESS types. Fixed to use JC/JNC.

2. upeep80/peephole.py did not translate 8080's JP (jump if plus) to Z80's
   JP P, conditional jump. Added translation for JP opcode.

**Fix applied**:
- Changed _emit_jump_on_false_16bit to use JC/JNC (carry-based) comparisons
- Changed _emit_jump_on_true_16bit to use JC/JNC (carry-based) comparisons
- Fixed _gen_comparison to use MOV A,L / ORA H instead of ORA B for GT/LE
- Added JP translation in upeep80: if opcode == "JP" -> "JP P,"


## Bug 3: FOR Loop with DJNZ Has Register Allocation Bug

**Status**: FIXED
**Severity**: High
**Location**: codegen.py (FOR loop code generation)

**Description**:
FOR loops that use the DJNZ optimization had two register B allocation issues:

1. DJNZ loop uses B as counter, but loop body may clobber B
2. Variable-bound byte loops saved index in B before evaluating bound,
   but bound expression evaluation could clobber B

**Example**:
```plm
DO I = 0 TO 4;
    X = Y + Z;  /* Addition clobbers B, corrupting loop counter */
END;
```

**Fix applied**:
1. Added PUSH B / POP B around the loop body in DJNZ loops to preserve
   the loop counter even when body operations clobber B
2. Rewrote variable-bound comparison to evaluate bound first into B,
   then load index into A, avoiding the B-clobbering issue


## Bug 4: Module-Level DATA Declarations Address Calculation

**Status**: TODO (may be design limitation)
**Severity**: Medium
**Location**: codegen.py or symbols.py

**Description**:
DATA declarations at module level cause incorrect address calculations.
Strings and data tables end up at wrong addresses.

**Example**:
```plm
DECLARE MSG DATA('Hello$');  /* At module level - broken */

MAIN: PROCEDURE;
    DECLARE MSG DATA('Hello$');  /* Inside procedure - works */
END MAIN;
```

**Current workaround**: Place all DATA declarations inside procedures.


## Bug 5: ADDRESS Subtraction in Loops May Have Issues

**Status**: FIXED (by Bug 2 fix)
**Severity**: Medium
**Location**: codegen.py (ADDRESS comparison)

**Description**:
Using ADDRESS variables as loop counters with decrement had issues.
When code like `DO WHILE ADDR > 0; ... ADDR = ADDR - 1; END;` was used,
the loop would not terminate correctly because ADDRESS comparisons were
using sign-based logic instead of unsigned carry-based logic.

**Example**:
```plm
DECLARE COUNT ADDRESS;
COUNT = 1;
DO WHILE COUNT > 0;
    /* do something */
    COUNT = COUNT - 1;
END;
```

**Fix**: This issue was fixed as part of Bug 2 (ADDRESS comparison fix).
The comparison now correctly uses carry-based unsigned comparison.
